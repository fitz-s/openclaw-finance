这也是为什么 peacetime 机会抓不住、暗流监测抓不稳、follow-up 又总是答非所问。当前 `custom_metric_compiler.py` 主要还是给 thesis/opportunity 附 `price_snapshot` 和 `option_flow_snapshot`；`finance_report_reader_bundle.py` 的 starter questions 只从前几个 object card 自动长出来；`finance_llm_context_pack.py` 的 `report_followup` 虽然口头上说“immutable bundle + selected handle rehydrate”，但 pack 本身仍然是 latest bundle 摘要；`finance_followup_answer_guard.py` 主要在 guard 结构和越权，没有把 `why / compare / challenge / scenario / sources` 变成不同的 evidence slice。你得到的是“对象层很强、证据层很弱”的系统，所以最后只能显得聪明，却不够准。([GitHub][4])

我这次只给一条主路径，不分散注意力：

# **OpenClaw Finance Information Dominance Stack**

副标题：**Source-to-Campaign Intelligence Fabric**

它不是“再加几个数据源”，也不是“再改一版 Discord UX”。它的目标是：**任何进入系统的高价值信息，都先成为可追溯、可重放、可比较、可积累的 evidence/claim/context-gaps，再被编进 peacetime scouts、undercurrents、campaign boards、follow-up slices。**
一句话的设计戒律是：

## **Never compress before provenance.**

在 provenance、lineage、event-time、rights、confidence 没固定之前，不允许把信息压成“summary”。

这条路不是凭空想象。公开可证的顶级机构/平台，真正的共性都在这里：Two Sigma 把“data as code”当成 resilient/adaptable data systems 与 ownership/quality metrics 的基础；BlackRock 明确用 originality、coverage/completeness、timeliness/latency、transparency/lineage 来评估数据；HRT 直接说如果 ingest 的 data fundamentally flawed，算法就 destined to fail；Point72 的 Market Intelligence 把 sourcing、compliance、engineering、research product 连成一条链；RavenPack 强调 relevance / novelty / impact；Dataminr 做的是百万级 public sources 的 early event detection + context；AlphaSense 的核心是巨量 premium docs、expert interviews、internal content 的 intelligence fabric；Bloomberg 强在把 portfolio analytics、risk、attribution、reporting 接成 decision console。它们外壳不同，但 operating model 一样：**source quality + point-in-time lineage + event/entity intelligence + portfolio translation**。([Two Sigma][5])

## Layer A — `Source Office`

这是第一层，也是最重要的一层。

### 你要做的不是“多接源”，而是建立 **六条 source lanes**

`market_structure`
价格、成交、ETF/指数、期权链/OI/skew、sector relatives、rates/fx/commodities。当前 `price_fetcher.py` 和 `options_flow_proxy_fetcher.py` 都只是 snapshot/proxy/fallback，这条 lane 至少要升级到 point-in-time、provider-agnostic、有 freshness SLA 的 market context，而不是继续把 yfinance snapshot 当成 quasi-truth。([GitHub][6])

`corp_filing_ir`
10-Q/10-K/8-K、earnings transcripts、guidance、IR press releases、investor presentations、calendar、insider trades、13F/13D、buyback/debt issuance、segment KPI。现在 `sec_discovery_fetcher.py` 只是 current atom feed 上的轻量 current-form discovery，这对真正的 investment intelligence 来说远远不够。([GitHub][7])

`real_economy_alt`
job postings、pricing、traffic、shipping/customs、power load、weather/disaster、web/app demand、geospatial、channel checks。公开材料里 Point72 Canvas、HRT、BlackRock 都强调 dataset 的 originality/coverage/timeliness 与 field/supply-chain research 的价值；这条 lane 正是 peacetime opportunity 和 undercurrent 的关键。([BlackRock][8])

`news_policy_narrative`
不是普通新闻摘要，而是 entity/event/novelty/impact intelligence。RavenPack 的 relevance/novelty/impact、Dataminr 的 earliest possible indications 和 multimodal public-source detection，都是这条 lane 该学的。当前 reviewer packet 已经显示你几乎全靠这一条 lane 在说话，但这条 lane 现在还太像“摘要新闻”，不像“事件 intelligence”。([RavenPack][9])

`human_field_private`
这里吸收的是**合规的私有研究**：expert transcripts、paid research、用户自己的 notes、field memo、渠道纪要。只能是 compliant internal/private，不碰 MNPI。Point72 公开材料把 compliance 明确绑定在这条链上，这一点不能模糊。([Point72][10])

`internal_private`
watch intent、portfolio attachment、capital buckets、historical outcomes、committee memos、old thread unknowns、own notes。外部平台做不到这条 lane 的深度，OpenClaw 反而在这里天然最强。

### promotion 逻辑不要再只看 ticker move

```python
def undercurrent_score(
    persistence: float,
    acceleration: float,
    cross_lane_confirmation: float,
    capital_relevance: float,
    contradiction_load: float,
    freshness_penalty: float,
) -> float:
    return (
        0.28 * persistence
        + 0.18 * acceleration
        + 0.24 * cross_lane_confirmation
        + 0.20 * capital_relevance
        - 0.15 * contradiction_load
        - 0.10 * freshness_penalty
    )

def promote_to_campaign(card: UndercurrentCard) -> bool:
    return (
        card.source_diversity >= 2
        and card.cross_lane_confirmation >= 0.45
        and card.contradiction_load <= 0.35
        and card.capital_relevance >= 0.50
        and card.persistence_score >= 0.55
    )
```

## Layer F — `Follow-up Committee Room`

现在 follow-up 的核心错误不是“上下文不够长”，而是“取上下文时没按动词和对象切片”。`finance_llm_context_pack.py` 和 `finance_followup_answer_guard.py` 现在更多是在做 generic rehydrate + structure guard，这不够。([GitHub][12])

### 强制新 contract：`verb + primary_handle (+ secondary_handle)`

```python
# scripts/finance_followup_context_router.py
VERB_SLICES = {
    "why": [
        "campaign_projection",
        "recent_claims",
        "source_health",
        "promotion_reason",
        "event_timeline",
    ],
    "challenge": [
        "countercase_memo",
        "invalidator_cluster",
        "contradiction_claims",
        "denied_hypotheses",
        "freshness_risks",
    ],
    "compare": [
        "capital_graph_slice",
        "displacement_case",
        "bucket_competition",
        "portfolio_attachment",
    ],
    "scenario": [
        "scenario_exposure_slice",
        "hedge_coverage",
        "crowding_risk",
        "linked_campaigns",
    ],
    "sources": [
        "source_atoms",
        "claim_lineage",
        "rights_and_redaction",
        "freshness_by_lane",
    ],
}

def build_followup_context(bundle, verb, primary, secondary=None):
    required = VERB_SLICES[verb]
    slice_bundle = bundle.select(required, primary=primary, secondary=secondary)
    missing = slice_bundle.missing_required()
    if missing:
        return {
            "status": "insufficient_data",
            "missing": missing,
            "context_gaps": bundle.lookup_context_gaps(primary, missing),
        }
    return slice_bundle
```

### 预烘焙 deep-dive cache

对 top campaigns，系统在 report 发出前就异步生成：

* `why_now card`
* `challenge card`
* `compare current book card`
* `scenario stress card`
* `sources freshness card`

这才是多 agent 的正确用法：不是拿来堆 prose，而是拿来堆 **anticipatory context**。

## 5. 关键实现细节，防止实现者跑偏

### 设计约束 1：**parallel-write before cutover**

第一阶段不要马上杀掉老 observation。
做：

* old normalized observation 继续写
* `EvidenceAtom` / `ClaimAtom` / `ContextGap` 并行写
* downstream 先只读新层做对照，不立刻切主链

这样可以避免一开始就把整个 hot path 炸掉。

### 设计约束 2：**reviewer/export 永远 sanitized；runtime 内部可以 richer**

你现在 reviewer packet 已经做了 redaction，这是对的；不能为了 reviewer 可见性把 raw snippets 暴露出去。正确做法是：

* raw vault internal only
* safe excerpts runtime-available if rights permit
* reviewer packets always derived-only

### 设计约束 3：**所有高影响判断都要能 trace 到 claim_ids**

你后面所有 report、board card、thread answer，至少要能内部映射：

`operator sentence -> claim_ids -> atom_ids -> source_ids`

这一步一旦成了，report density 才有资格被量化。

### 设计约束 4：**unknown 必须对象化**

不允许只写“to verify”。
必须写成 `ContextGap`，并且要能说：

* missing lane
* why load-bearing
* cost of ignorance
* suggested sources

### 设计约束 5：**peacetime 不是副产品**

`PACKET_UPDATE_ONLY` 至少要能驱动：

* `Peacetime Board` mutation
* `UndercurrentCard` score update
* `CampaignProjection` stage transition
* deep-dive cache refresh

### 真正的 acceptance criteria

1. reviewer packet 能做 **exact report-time replay**，不再只是 current sanitized snapshot

## 7. rollout / rollback / monitoring

### rollout

**Phase 1**
parallel write `EvidenceAtom / ClaimAtom / ContextGap`，不动主 hot path。

**Phase 2**
`Undercurrent Engine` + `Peacetime Board` 上线，但只做 board mutation，不影响 wake priority。

**Phase 3**
`CampaignProjection` 成为 Discord 主消费面；artifact markdown 保留。

**Phase 4**
follow-up 切到 verb-specific slices + deep-dive cache。

**Phase 5**
再考虑让 undercurrent promotion 影响 wake priority。

### rollback

* 关闭 atom/claim consumers，保留旧 normalized observation
* board fallback 到当前 `discord_primary_markdown`
* follow-up fallback 到 current bundle digest
* exporter 继续导出 sanitized reviewer packets，不受新层影响

### monitor

* `source_coverage_score`
* `freshness_sla_breach_rate`
* `source_diversity_per_campaign`
* `claims_per_1000_chars`
* `context_gap_rate`
* `context_gap_closure_time`
* `peacetime_to_live_conversion`
* `undercurrent_false_positive_rate`
* `followup_grounding_failure_rate`
* `raw_snippet_export_count == 0`

## 8. 现在明确不做的

不做 standalone terminal / dashboard app。
不做 execution adapter。
不做 HFT / co-lo / Level-3 arms race——IMC/Optiver 那是另一种 horizon，你要吸收的是它们对 source quality 和 market data discipline 的 operating model，不是复制它们的微秒栈。IMC 公开披露自己直接用 direct market data feeds、daily process petabytes；Optiver 公开讨论 Level-3 historical data 的 sourcing 难题，这说明真正高频机构的数据栈成本和目标都完全不同。对你现在这套 review-only、Discord-native、campaign-centric 系统，正确目标是 **比当前 snapshot/proxy 高一个数量级的 point-in-time context**，而不是假装做成 market maker。([IMC Trading][20])

---

最后把最关键的一句说死：

**你现在最需要的，不是再造一个更华丽的金融报告器，而是建立一个真正的 `information office`：上游用 lane-specific source registry 和 evidence atoms 把世界装进系统，中间用 claim graph / context gaps / undercurrents 把世界压成可决策结构，下游再用 campaign boards 和 verb-specific committee room 让你在 Discord 里消费它。**

做到这一步，OpenClaw Finance 才不再是在“对低质摘要做更强推理”；它会开始拥有自己的 **information edge**。
