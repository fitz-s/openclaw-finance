# Finance Report Contract

Finance reports must be rendered from the typed decision chain:

```text
ContextPacket
  -> WakeDecision
  -> JudgmentEnvelope
  -> product report envelope
  -> product validator
  -> decision log
  -> delivery safety gate
```

The active user-visible report source is:

- `finance/state/finance-decision-report-envelope.json`

The active report must be generated by:

- `finance/scripts/finance_decision_report_render.py`

The active product validator is:

- `finance/scripts/finance_report_product_validator.py`

## Input Authority

The report renderer may read typed packet, wake, judgment, validation, portfolio, watchlist, price, options-flow, SEC-semantic, broad-market, and scanner state artifacts. It must not read raw Flex XML, raw news text, account identifiers, or secrets.

The renderer must treat unavailable facts as explicit non-claims. It must not narrate unavailable or stale data as fresh.

## Compatibility View

`finance/state/report-input-packet.json` is not the active report cognition substrate. It is a compatibility mirror for old validators/audits and must be marked:

- `compatibility_view_only=true`
- `must_not_be_used_as_cognition_source=true`

Any prompt, job, or script that treats `report-input-packet.json` as the only active report substrate is stale and must be updated or marked compatibility-only.

## Product Obligations

The product report must:

- always collect and analyze the core macro triad: Gold, Bitcoin, and SPX/SPY direction; unavailable data must be explicit, not silently omitted
- present a clear decision context without execution language
- prioritize opportunity expansion and watchlist/flow context before holdings narration
- suppress raw internal phrases such as `thresholds not met`
- suppress raw provenance logs such as excessive `ev:*`, `packet_hash`, and `judgment_id` in user-visible markdown
- retain machine provenance in the envelope and decision log
- explain stale or unavailable data as data-quality constraints, not as market conclusions
- pass `finance_report_product_validator.py` before delivery

## LLM Rewrite Boundary

If an LLM rewrite is ever used, it may only rewrite the product markdown from an already built product envelope. It may not add facts, alter evidence references, alter policy version, or bypass product validation and delivery safety.

## Failure Rule

If product validation or delivery safety fails, the OpenClaw report orchestrator must output a health/system status message only. It must not output market judgment, watchlist moves, portfolio impact, ticker changes, news conclusions, or trade suggestions.
